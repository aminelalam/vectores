# RAG Semantic Search

Ask questions about your own PDFs and get concise answers with traceable sources.

**Highlights**
- Hybrid retrieval: dense embeddings + local BM25, fused with Reciprocal Rank Fusion.
- Query expansion (singular/plural + ES/CA term variants) and fuzzy matching for typo tolerance.
- Coverage-aware reranking and numeric boost for quantity questions.
- Strict RAG prompting: answers are grounded in retrieved context only, with source refs.
- Multiple interfaces: FastAPI API, lightweight Web UI, and optional Streamlit app.
- End-to-end ingestion pipeline from PDFs to Pinecone.

**Overview**
This project implements a Retrieval-Augmented Generation (RAG) search system for PDF documents. It combines
vector search in Pinecone with a local lexical index (BM25), then reranks results based on query coverage and
numeric relevance. Answers are generated by an LLM (OpenAI or Ollama) using only retrieved context and are
returned with source references. The generation prompt expects Spanish output and understands Spanish and
Catalan terminology.

**What I Built and Why (Personal)**
I built this project to answer real questions from real PDFs without losing trust in the results. Here is what I chose and why:
- I built a PDF ingestion pipeline that cleans text, chunks it, and builds embeddings so documents are searchable and the process is repeatable.
- I chose Pinecone for vector indexing so semantic search stays fast even as the document set grows.
- I added a local BM25 index because exact keyword matches still matter and dense embeddings alone can miss them.
- I use Reciprocal Rank Fusion to blend dense and lexical results so one signal never overwhelms the other.
- I implemented query expansion and fuzzy matching to handle singular/plural forms, ES/CA variants, and small typos.
- I rerank by query coverage and numeric signals so the top chunks are the ones most likely to contain the answer, especially for quantity questions.
- I enforce a strict RAG prompt that uses only retrieved context and cites sources so answers are grounded and auditable.
- I exposed the system through FastAPI, a simple Web UI, and an optional Streamlit app so both developers and non-technical users can work with it.

**What It Enables**
- Ask natural-language questions over your PDFs and receive cited answers.
- Upload new PDFs and re-ingest without rewriting code.
- Swap LLM providers between OpenAI and Ollama with a single env change.
- Tune retrieval depth, chunking strategy, and context size for your domain.

**Architecture**
- Ingestion: `PDF -> clean text -> chunks -> embeddings -> Pinecone`.
- Retrieval: hybrid dense + lexical (BM25), merged with Reciprocal Rank Fusion.
- Rerank: query coverage boost + numeric boost for quantity questions.
- Generation: LLM (`openai` or `ollama`) using retrieved context only.
- Interfaces: FastAPI (`/ask`, `/upload`, `/documents`, `/ingest`, `/health`), Web UI served at `/`, optional Streamlit frontend.

**Requirements**
- Python 3.10+
- A Pinecone account and API key
- One LLM provider: OpenAI API key or local Ollama instance

**Quickstart**
```bash
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install -r requirements.txt
```

Create a `.env` file (see Configuration), then run the API:

```powershell
.\start.ps1
```

If PowerShell script execution is blocked:

```cmd
start.cmd
```

Run on a custom port:

```powershell
.\start.ps1 -Port 8010
```

Skip environment checks (not recommended):

```powershell
.\start.ps1 -SkipChecks
```

You can also run Uvicorn directly:

```bash
uvicorn api.main:app --reload --host 0.0.0.0 --port 8010
```

**Configuration**
Create a `.env` file in the project root. The startup script runs `scripts/verify_setup.py`, which requires
all of the variables below (unless you use `-SkipChecks`).

```env
# Pinecone
PINECONE_API_KEY=YOUR_PINECONE_KEY
PINECONE_INDEX=rag-semantic-search
PINECONE_NAMESPACE=default
PINECONE_CLOUD=aws
PINECONE_REGION=us-east-1

# LLM provider (openai or ollama)
LLM_PROVIDER=openai
OPENAI_API_KEY=YOUR_OPENAI_KEY
OPENAI_MODEL=gpt-4o-mini
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.1
LLM_TIMEOUT_SECONDS=120

# Retrieval and embeddings
EMBEDDING_MODEL=all-MiniLM-L6-v2
RAG_TOP_K=5
RAG_MAX_CONTEXT_CHARS=12000

# Ingestion
INGEST_CHUNK_SIZE=1000
INGEST_CHUNK_OVERLAP=200
INGEST_BATCH_SIZE=64
UPLOAD_MAX_SIZE_MB=40
```

Notes:
- `LLM_PROVIDER=openai` requires `OPENAI_API_KEY` and `OPENAI_MODEL`.
- `LLM_PROVIDER=ollama` requires `OLLAMA_BASE_URL` and `OLLAMA_MODEL`.
- If you work primarily in Spanish/Catalan, consider a multilingual embedding model.

**Ingestion Pipeline**
Place PDFs in `data/raw/`, then run:

```bash
python scripts/ingest_all.py
```

This runs: extract text -> chunk -> index in Pinecone. By default, indexing clears the target namespace to
avoid stale vectors. For cumulative indexing, run:

```bash
python scripts/index_pinecone.py --append
```

The local BM25 index is built from `data/chunks/chunks.jsonl`. If that file does not exist, lexical retrieval
is skipped.

**API**
- `POST /ask` - Ask a question and get a source-cited answer.
- `POST /upload` - Upload PDF files.
- `POST /ingest` - Run the ingestion pipeline.
- `GET /documents` - List PDFs in `data/raw/`.
- `GET /health` - Health status and Pinecone index name.
- `POST /engine/reload` - Reload the RAG engine.

Example request:

```bash
curl -X POST http://localhost:8010/ask \
  -H "Content-Type: application/json" \
  -d "{\"query\":\"Que dice el informe sobre inflacion?\",\"top_k\":5}"
```
